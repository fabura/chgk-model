# θ̄-aware question initialisation

## Motivation

`docs/noisy_or_init_experiments.md` описывает первый шаг —
коррекцию init на размер команды (`b = log(n) − log(−log(1 − p))`),
которая убрала систематическое занижение `b` на `log(6) ≈ 1.79`
и дала `−0.0088` logloss на бэктесте.

После этого фикса остался **второй уровень** той же ошибки. Формула
`b = log(n) − log(−log(1 − p))` всё ещё неявно предполагает, что
средняя команда играет **при θ = 0**. На сильнорейтинговых турнирах
(Высшая лига, Гран-при, Гембец) это не так: средняя команда там
имеет `θ̄ ≈ +0.5…+1.0`. По noisy-OR честный init для команды из
`n` игроков средней силы `θ̄`:

```
1 − exp(−n · exp(−b + a·θ̄)) = p_take      (a = 1 при init)
⇒ b = log(n) + θ̄ − log(−log(1 − p_take))
```

Без `+θ̄` модель снова инициализирует `b` на `θ̄` единиц ниже
правды — и через ту же gauge-неопределённость остаточная ошибка
утекает в θ ветеранов команды-победителя.

Эмпирически это видно на конкретных турнирах. Diagnostic-сравнение
(до/после) на трёх днях Высшей лиги Москвы (`scripts/diagnostic_compare.py`):

| турнир | mean Δ_old | mean Δ_new | swing |
|---|---:|---:|---:|
| ВЛМ день 1 | −4.5 | +0.3 | +4.8 |
| ВЛМ день 2 | −5.3 | +0.2 | +5.5 |
| ВЛМ день 3 | −5.8 | +0.3 | +6.1 |

(`Δ = actual − expected`; отрицательный Δ = модель переоценивает
команду и стягивает θ её игроков вниз.) Раньше **все 11 команд** в
каждом из трёх дней получали `expected ≈ 5–6` лишних взяток,
сейчас `expected` практически равен среднему фактическому.
На уже-калиброванном Гран-при текстильной столицы (`+0.7 → +1.3`)
изменение незначительное — фикс симметричен и не ломает нормально
работающие пакеты.

## Implementation

`Config.theta_bar_init: bool = True` (новый дефолт),
`Config.theta_bar_min_games: int = 3`.

`QuestionState.init_from_take_rate` принимает дополнительный
аргумент `theta_bar: float | None`. Когда передан — формула:

```
b_i = log(n) + θ̄ − log(−log(1 − p_take))
```

Когда `theta_bar is None` — фолбэк на noisy-OR-only формулу
`b = log(n) − log(−log(1 − p))` (т.е. чистый Round 1).

`run_sequential` для каждого ещё неинициализированного вопроса
собирает по командам, видевшим этот вопрос на этом турнире,
**среднее θ зрелых игроков** (`games >= theta_bar_min_games`):

```python
mature = players.games[pids] >= theta_bar_min_games
if mature.any():
    q_theta_bars[qi].append(float(players.theta[pids[mature]].mean()))
...
theta_bar = mean(q_theta_bars[qi]) if q_theta_bars[qi] else None
```

Новички исключены из θ̄ потому, что их θ доминируется
`cold_init_theta = −1.0` и систематически занизила бы оценку (особенно
в начале сезонов, когда новички приходят пачками). Если в команде
вообще нет зрелых игроков — наблюдение не вносит вклада в θ̄;
если ни одно наблюдение не дало θ̄ — init фолбэчит на формулу
без поправки.

CLI: `--theta-bar-init / --no-theta-bar-init`,
`--theta-bar-min-games N`.

## Backtest

20 % time-split hold-out, full DB, все остальные дефолты — `t6`
2026-04 lean + noisy-OR init.

| config | logloss | offline | sync | async | AUC |
|---|---:|---:|---:|---:|---:|
| `noisy_or_only` (Round 1 финал) | 0.5182 | 0.4791 | 0.5167 | 0.5396 | 0.8333 |
| `theta_bar_init` (без re-tune) | 0.4950 | 0.4563 | 0.4893 | 0.5238 | 0.8395 |

Сам θ̄-фикс даёт **−0.0232 logloss** (~4.5%) и **+0.0062 AUC**
без переподбора шагов — чистый init-фикс.

## Re-tune (round 2)

Гипотеза: с ещё более точным `b_init` модель «недокручивает»
θ-обновления. 5-параметровый coord-descent (`eta0`, `w_sync`,
`w_online_questions`, `eta_size`, `eta_pos`) — single-knob, потом
combos. Полная таблица — `results/exp_theta_bar_retune.csv`.

Single-knob выводы:

* **`eta0` хочет сильно выше**, монотонно вплоть до 0.15. Пик
  плоский на 0.15…0.20; 0.25 уже регрессирует. Победитель — 0.15.
  Контринтуитивно, но логично: с θ̄-аware `b`-init итерации θ
  работают по чистому сигналу (а не догоняют системный байас в `b`),
  и могут позволить себе шаг в ~3.5× больше старого. На асинке
  улучшение тоже монотонное (0.5238 → 0.5172).
* **`w_sync` хочет выше** (0.5 → 1.0 даёт −0.0022), но в combo
  с `eta0=0.15` лучше уже его не трогать — combos с `eta0=0.15` +
  `ws>0.5` либо равны, либо хуже single `eta0=0.15` + `ws=0.5`.
  Шаги по θ при `eta0=0.15` уже агрессивные сами по себе;
  `w_sync>0.5` начинает overshoot’ить sync-наблюдения.
* **`w_online_questions`, `eta_size`, `eta_pos`** — плоские, signal
  ниже шума (0.0001). Оставлены без изменений.

## Final defaults (`Config`)

```
theta_bar_init        = True       # was: implicit False / N/A
theta_bar_min_games   = 3
eta0                  = 0.15       # was 0.04 (Round 1 picked 0.04 for noisy-OR-only)
```

Все остальные `Config` параметры — без изменений по сравнению с
Round 1 (`noisy_or_init=True`, `eta0=0.04` → теперь `0.15`,
`w_sync=0.5`, `w_online_questions=0.15`, `eta_size=eta_pos=0.001`).

## Reproduce

```bash
# Single config
.venv/bin/python scripts/exp_theta_bar_retune.py eta0_0.15

# All ~25 configs (sweep + combos), ~3 hours
.venv/bin/python scripts/exp_theta_bar_retune.py --list \
  | xargs -I {} .venv/bin/python scripts/exp_theta_bar_retune.py {}

# Compare new defaults vs no-θ̄ on hold-out
.venv/bin/python -m rating --mode cached --cache_file data.npz --backtest                          # new defaults
.venv/bin/python -m rating --mode cached --cache_file data.npz --backtest --no-theta-bar-init      # only θ̄ disabled

# Per-tournament before/after diff (requires two DuckDB snapshots)
cp -p website/data/chgk.duckdb website/data/chgk.duckdb.before_thbar
# … re-train + re-build website/data/chgk.duckdb …
.venv/bin/python scripts/diagnostic_compare.py \
    --old website/data/chgk.duckdb.before_thbar \
    --new website/data/chgk.duckdb \
    13345 13104 13041 12469
```

`results/exp_theta_bar_retune.csv` — полная таблица бэктеста по
всем конфигам (включая нарезку по hardness-квартилям θ̄).

## Notes / caveats

* θ̄ берётся из **pre-tournament** θ (того, что висит в
  `players.theta` на момент входа в турнир). Это значит, что для
  самого первого турнира в истории команды-новичка θ̄ может быть
  смещено вниз (новичков много). На практике это не проблема —
  большинство команд либо стабильны (старые игроки), либо вовсе
  не дают вклада в θ̄ (все игроки `games < 3`).
* `theta_bar_min_games = 3` — мягкий порог; sweep на 1/3/5 не делал,
  но эвристически 3 разумен (после 3 турниров θ начинает приближаться
  к стабильному значению, ср. `cold_init_theta = −1.0` и
  `games_offset = 0.25`).
* На «легких» турнирах (online quizzes, async с `θ̄ ≈ 0`) поправка
  близка к нулю — фикс работает только там, где он нужен.
* Дополнительный sanity check: дельта θ для конкретных игроков
  на конкретных турнирах теперь **симметрична**. Команда «Борский
  корабел» (с Александром Либером) на Высшей лиге Москвы день 3
  взяла 30 вопросов при `expected_new=21.9`; раньше было
  `expected_old=26.9` (т.е. модель занижала превосходство сильной
  команды). Соответственно теперь её игроки получают положительную
  Δθ, как должно быть, а не отрицательную, как раньше.
* Top-1 после полного прогона — Семушин Иван (1077 игр,
  θ=0.998), что реалистично для сильнейшего активного игрока ЧГК.
  Александр Либер по результатам разбора выходит на ~#52 —
  выше прежнего ~#45 не сдвинулся ощутимо, но это и не
  обязательно: цель была убрать систематическую утечку через
  pack-level байас, а не сделать кого-то конкретного №1.
  Эффект сильнее всего на ветеранах команд с высокой
  посещаемостью «Вышки» / «Гран-при» — на них модель раньше
  систематически отнимала θ за «недовыполнение завышенных
  ожиданий» при том, что они эти турниры выигрывали.
