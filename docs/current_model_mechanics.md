> Этот документ описывает старую офлайновую ветку (`train.py`,
> `model.py`, старые скрипты экспорта) и больше не является источником
> истины для текущего sequential-кода в `rating/`.
>
> Для текущей модели см.:
>
> - `README.md`

Этот документ описывает, как была устроена старая офлайновая модель в
`train.py`, `model.py`, `data.py` и старых скриптах экспорта.

## 1) Сила игрока, сложность вопроса, селективность

Базовые параметры:

- `theta_k` — сила игрока `k`
- `b_i` — сложность вопроса `i`
- `a_i = exp(log_a_i)` — селективность (дискриминация) вопроса `i`

Для игрока `k` и вопроса `i`:

`lambda_ik = exp(-b_eff_i + a_i * theta_k)`

где `b_eff_i` — эффективная сложность вопроса (см. раздел про сложность турнира).

Для команды:

- `lambda_sum = sum_k lambda_ik`
- `p_take = 1 - exp(-lambda_sum * exp(team_size_bias[team_size]))`

То есть вклад игроков **аддитивный** по `lambda`.

## 2) Как учитывается сложность турнира и тип турнира

Для каждого вопроса в данных есть:

- `tournament_dl` — сложность турнира (берется из `true_dls`, если нет — fallback из `ndcg`)
- `tournament_type` — тип турнира: `0=очник`, `1=синхрон`, `2=асинхрон`

В модели есть:

- `tournament_type_bias[3]` — аддитивный сдвиг сложности по типу турнира,
- `tournament_dl_scale[3]` — коэффициент при `true_dl` по типу,
- transform `f(dl)` для `tournament_dl`:

- `linear` (по умолчанию): `f(dl)=dl`
- `log1p`: `f(dl)=log1p(alpha*max(dl,0))`
- `power`: `f(dl)=sign(dl)*|dl|^gamma`

Эффективная сложность вопроса:

Сначала `tournament_dl` нормируется внутри типа турнира (`z-score by type`), затем применяется transform:

`dl_z_i = (tournament_dl_i - mean_dl[type_i]) / std_dl[type_i]`

`b_eff_i = b_i + tournament_type_bias[type_i] + tournament_dl_scale[type_i] * f(dl_z_i)`

## 3) Инициализация параметров

Если обучение не из чекпоинта:

- `b` инициализируется по эмпирической доле взятий вопроса:
  - высокая доля взятий -> меньшая сложность,
  - низкая доля взятий -> большая сложность.
- `log_a` может инициализироваться через корреляцию `(taken, team_strength)` по вопросу.
- `theta` стартует из небольшого случайного распределения, затем сдвигается при инициализации.

Если `--resume`, параметры загружаются из чекпоинта.

## 4) Фильтры данных (что попадает в обучение)

### 4.1 Фильтр по дате турнира

`--min_tournament_date` (по умолчанию `2015-01-01`): в выборку берутся только турниры с `start_datetime >= date` (или `NULL`).

### 4.2 Фильтр по числу игр игрока

`--min_games_in_dataset` (по умолчанию `10`):

- при загрузке из БД игроки с `<N` игр удаляются из состава (как "пустой стул"),
- команда исключается только если после удаления таких игроков состав стал пустым,
- при загрузке из кэша применяется та же логика, затем выполняется переиндексация игроков.

### 4.3 Заморозка слабонаблюдаемых игроков

Игроки с числом игр `< MIN_GAMES_FOR_TRAIN` (сейчас 30) получают:

- `theta = 0`,
- градиент по `theta` зануляется,
- после шага оптимизатора `theta` снова принудительно ставится в 0.

## 5) Регуляризация и идентифицируемость

### 5.1 L2-регуляризация

В loss добавляется L2 для `theta`, `b`, `log_a`.
Вес регуляризации задается через `--reg_theta`, `--reg_b`, `--reg_log_a`.

Для игроков и вопросов используются observation-normalized веса `~ 1/sqrt(count)`:

- меньше наблюдений -> сильнее штраф.

### 5.2 Центрирование (identifiability)

После каждого шага:

- `theta` центрируется (среднее вычитается; предпочтительно по опытным игрокам 500+ игр),
- `b` стабилизируется вокруг `mean_b=0` через отдельный штраф.

## 6) Почему сильные известные игроки могут быть низко в рейтинге

Ниже причины, которые прямо следуют из текущей реализации:

1. **Аддитивная агрегация по команде**  
   Модель хуже отражает сценарии "один очень сильный игрок против полной команды". Это известное ограничение и описано в `docs/interpretation.md`.

2. **Сильная регуляризация и центрирование `theta`**  
   Оба механизма тянут параметры к центру, особенно если по игроку не очень много наблюдений в выбранном срезе.

3. **Жесткие фильтры по данным**  
   После `min_tournament_date`, `min_games_in_dataset` и freeze-маски фактический набор игр может заметно отличаться от "исторической силы" игрока.

4. **Сложность турнира корректирует именно `b` вопросов**  
   Если эта часть недообучена/переобучена на конкретный тип турниров, вклад может перекладываться между `b`, `a` и `theta`.

5. **Экспорт `players.csv` по умолчанию дополнительно режет по числу игр**  
   В конце обучения экспорт учитывает `--export_min_games` (по умолчанию 30) и `--min_tournament_date`.

## 7) Что именно уходит в `players.csv` сейчас

По умолчанию в конце `train.py`:

- пишется `results/players.csv`,
- формат: `player_id,theta,train_games,train_samples`,
- если доступна БД и `--export_min_games > 0`, то экспортируются только игроки с минимум `export_min_games` турнирами (с учетом `--min_tournament_date`).

Та же логика фильтрации поддержана в `scripts/export_checkpoint_results.py`.
