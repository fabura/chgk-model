# Noisy-OR-aware question initialisation

## Motivation

A user (Юлия в чате друзей, апрель 2026) noticed that strong players
who consistently play hard tournaments — Александр Либер as the
canonical example — were sitting much lower in the rating than her
intuition predicted. On Высшая лига Москвы 2-й игровой день, on
Гран-при Гембеца, on отбор в Иваново и т.д. Либер, Попов и
другие сильные игроки получали **отрицательное** изменение θ —
несмотря на то, что эти турниры они выигрывали или брали почти
поровну с лидерами.

Беглый разбор показал, что это симптом, а не отдельная аномалия:

* На сложных турнирах **все** команды (включая победителя)
  систематически брали меньше, чем `expected_takes` модели — то
  есть модель завышала ожидания на «сложных» пакетах.
* На завышенных ожиданиях даже у победителей `actual − expected < 0`,
  и модель аккуратно стягивает их θ вниз.
* Сильные игроки, играющие много сложных турниров, накапливают это
  стягивание непропорционально — и опускаются в рейтинге ниже
  «нонеймов», играющих лёгкие пакеты.

## Diagnosis

Корень проблемы — формула инициализации сложности вопроса
`b_i` в `QuestionState.init_from_take_rate`. Старая формула:

```
b_i  =  −log(p_take_i)
```

неявно предполагает, что вопрос пытается взять **один** игрок при
θ = 0. В нашей же модели вопрос берёт команда из `n` игроков, и
noisy-OR с `n` независимыми источниками даёт:

```
p_take = 1 − exp(−n · exp(−b))    (при θ = 0, a = 1)
```

То есть «честный» init для команды среднего размера `n`:

```
b_i  =  log(n)  −  log(−log(1 − p_take_i))
```

Старая формула систематически **занижает** `b` на величину
`log(n)`. Для типичной команды из 6 игроков это сдвиг на
`log(6) ≈ 1.79` единиц. Это огромная ошибка: на «лёгких» вопросах
(`p ≈ 0.7`) старый init давал `b ≈ 0.36`, новый — `b ≈ 1.94`.

Внутри одного турнира SGD, конечно, пытается корректировать `b`
вверх, но за `~36` наблюдений (количество команд × 1 question)
не успевает закрыть разрыв полностью; **остаточная ошибка** в
`b` через
*gauge-неопределённость* noisy-OR (`θ → θ − Δ`, `b → b + a·Δ` —
team-level вероятности инвариантны) утекает в `θ` игроков.
Особенно болезненно это для ветеранов с маленьким адаптивным
шагом `η = η0 / √(games_offset + games)`: они почти не двигают
свою θ внутри турнира, но коллективное смещение на pack-level
оказывается полностью на их балансе.

## Implementation

`Config.noisy_or_init: bool = True` (новый дефолт).
Когда флаг включён, `QuestionState.init_from_take_rate` принимает
дополнительный параметр `team_size_avg: float = 1.0`:

```python
def init_from_take_rate(self, idx, take_rate, team_size_avg=1.0):
    r = clip(take_rate, 1e-6, 1 - 1e-6)
    n = max(team_size_avg, 1.0)
    if n <= 1.0:
        self.b[idx] = -log(r)              # legacy, identical
    else:
        self.b[idx] = log(n) - log(-log(1 - r))
    self.log_a[idx] = 0.0
    self.initialized[idx] = True
```

В `run_sequential` для каждого ещё неинициализированного
`canonical_q_idx` собирается среднее по командам, видевшим этот
вопрос в этом турнире, и передаётся в `init_from_take_rate`:

```python
n_avg = sum(team_sizes_for_q) / len(team_sizes_for_q)
questions.init_from_take_rate(qi, take_rate, team_size_avg=n_avg)
```

Это **только** init — никакой post-hoc коррекции. Дальнейший SGD
работает как обычно.

CLI-флаг `--noisy-or-init / --no-noisy-or-init` для аблэйшена.

## Backtest

20 % time-split hold-out, full DB (`data.npz`), все остальные
дефолты — `t6` 2026-04 lean.

| config | logloss | offline | sync | async | AUC |
|--------|--------:|--------:|-----:|------:|----:|
| `legacy_baseline` (noisy_or_init=False, t6 defaults) | 0.5270 | 0.5075 | 0.5224 | 0.5446 | 0.8158 |
| `noisy_or_baseline` (noisy_or_init=True, t6 defaults) | 0.5285 | 0.4878 | 0.5288 | 0.5474 | 0.8329 |
| **`combo_full`** (new defaults) | **0.5182** | **0.4791** | **0.5167** | **0.5396** | **0.8333** |

Без переподбора шагов чистый init-фикс даёт смешанный сигнал:
**offline-бакет (где как раз живёт «Вышка» / «Гембец») улучшается
на −0.020 logloss**, AUC по всем модам растёт на ~+0.017, но
overall logloss слегка ухудшается, потому что старые `eta0=0.07`,
`w_sync=0.7`, `w_online_questions=0.30`, `eta_size=eta_pos=0.005`
были «подкручены» именно под старую динамику b.

## Re-tune

5-параметровый coord-descent: `eta0`, `w_online_questions`,
`w_sync`, `eta_size`, `eta_pos`. Сначала single-knob (3 значения
вокруг старого дефолта), потом 8 комбо-конфигов из лучших
single-knob.

Single-knob выводы:

* **`eta0` хочет ниже**. 0.04 — оптимум по logloss; 0.03
 практически такой же. Старый дефолт 0.07 был «подкручен» под
 медленное движение `b` — теперь, когда `b` стартует на месте,
 такой агрессивный шаг по θ оказывается избыточным и даёт
 переобучение хвоста новичков.
* **`w_sync` хочет ниже**. 0.5 — оптимум; 0.4 чуть хуже. Сильный
 синхрон-вес тоже компенсировал плохой init: при правильном init
 новых турниров не нужно так интенсивно их «доучивать».
* **`w_online_questions`, `eta_size`, `eta_pos`** — почти
 плоский, эффект пренебрежимо мал, но монотонный в сторону
 уменьшения. 0.15 / 0.001 / 0.001 чуть-чуть лучше старых
 значений и хорошо сочетаются с понизившимися основными шагами.

Комбинации все сильно бьют любую single-knob. `combo_full` с
самыми низкими значениями всех пяти — лучший: даёт `−0.0088`
logloss и `+0.0175` AUC относительно legacy. Все три мода
улучшились одновременно (что важно — нет торга
offline-vs-async).

## Final defaults (`Config`)

```
noisy_or_init       = True       # was: implicit False / N/A
eta0                = 0.04       # was 0.07
w_sync              = 0.5        # was 0.7
w_online_questions  = 0.15       # was 0.30
eta_size            = 0.001      # was 0.005
eta_pos             = 0.001      # was 0.005
```

Все остальные знаки `Config` (rho, cold-start, recenter, solo
channel, …) — без изменений.

.venv/bin/python scripts/exp_noisy_or_init_retune.py --list \
  | xargs -I {} .venv/bin/python scripts/exp_noisy_or_init_retune.py {}

.venv/bin/python -m rating --mode cached --cache_file data.npz --backtest                                # new defaults

## Notes / caveats

* Init «правильный» только в смысле noisy-OR при θ = 0. Реальное
 распределение θ участников непустое; точная компенсация
 требовала бы знать априорное распределение силы команд на этом
 турнире, чего у нас в момент init нет. Эмпирически линейная
 поправка `log(n)` даёт почти всё улучшение (без неё SGD
 «доучивает» оставшийся остаток сам).
* Re-centering (gauge pinning раз в год) **не отменяет** этой
 утечки — он работает на уровне популяции, а описанная утечка
 происходит локально внутри одного турнира.
* Эффект особенно силён на «hard» турнирах (низкий `p_take` ⇒
 большой относительный недосчёт `b`). На лёгких пакетах
 `p_take ≈ 0.9` старая и новая формулы дают близкие значения.
* Структурное изменение init также убрало занижение θ у
 ветеранов сильных команд (тот самый кейс, с которого начался
 разбор). Дополнительные ранки/значения — после следующего
 полного `python -m rating` прогона; здесь они не приводятся,
 чтобы не путать «бэктест-метрики на hold-out» с
 «production-θ на текущей дате».
